## 2024-04-01 - [Missing Content-Security-Policy Header]
**Vulnerability:** The dashboard and web endpoints were lacking a Content-Security-Policy (CSP) header, relying only on X-XSS-Protection.
**Learning:** The application extensively uses inline scripts and styles within its HTML templates (e.g., `handler/dashboard.go`), meaning a strict CSP without `unsafe-inline` would break the UI.
**Prevention:** Added a baseline CSP (`default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline'; img-src 'self' data:;`) to `secureHandler` in `main.go`. In the future, refactoring inline scripts/styles into external files would allow for a stricter, more secure CSP.
## 2023-10-24 - [Plaintext Password Storage in RBAC]
**Vulnerability:** RBAC users' passwords were created and validated in plaintext using `subtle.ConstantTimeCompare`, exposing credentials if the dashboard config file is read.
**Learning:** External dependencies (like `golang.org/x/crypto/bcrypt`) cannot be downloaded due to strict network restrictions during builds. To securely store passwords without new dependencies, standard library functions (`crypto/sha256` + `crypto/rand`) must be used to create salted hashes, combined with a fallback mechanism to support pre-existing plaintext credentials from the environment.
**Prevention:** All new components that deal with authentication should use a `salt:hash` strategy utilizing standard library cryptography. Avoid using plain strings for passwords and handle backward compatibility safely to avoid breaking administrative lockouts.
## 2026-04-15 - [Missing nosniff header in direct JSON encoding]
**Vulnerability:** Some API endpoints were writing JSON responses directly using `json.NewEncoder(w).Encode(resp)` or similar methods. By not using the centralized `httputil.WriteJSON` helper or `http.Error`, these responses omitted the `X-Content-Type-Options: nosniff` header. This could potentially allow browsers to perform MIME-sniffing, presenting a security risk.
**Learning:** Even though there is a centralized function to handle JSON responses safely (`httputil.WriteJSON` which explicitly adds the `nosniff` header), it is easy to inadvertently bypass it by manually encoding JSON straight to the `http.ResponseWriter`.
**Prevention:** Always use `httputil.WriteJSON` to send JSON responses instead of `json.NewEncoder` to ensure all necessary security headers are included. Avoid writing responses manually unless strictly needed (e.g., streaming).

## 2024-05-31 - [Stored XSS via innerHTML in dashboard RBAC view]
**Vulnerability:** In `handler/dashboard.go`, the `rbacLoadUsers()` function was rendering dynamic user data (`u.username`, `u.role`) directly into HTML using `.innerHTML` without proper escaping. This allowed a malicious admin or someone able to inject data into the config storage to execute an XSS attack in the context of the dashboard view. Furthermore, string concatenation in inline event handlers like `onclick="rbacDeleteUser(' + u.username + ')"` exposed the app to attribute escape vulnerabilities.
**Learning:** Even internal admin tools mapping over data arrays to build HTML strings must properly escape every variable. Direct string concatenation into inline `onclick` attributes is highly prone to quoting bugs, breaking the application or leading to attribute breakouts.
**Prevention:** Always use `escHtml()` on string properties before rendering them inside `.innerHTML` strings. To handle data with quotes in inline event handlers, set a `data-*` attribute on the element (e.g. `data-user="safeValue"`) and retrieve it using `this.getAttribute('data-user')` instead of directly injecting the variable into the `onclick` handler.
## 2024-05-31 - [Stored XSS via incomplete escaping in JSON Syntax Highlighting (`colorizeJSON`)]
**Vulnerability:** The `colorizeJSON` function in the dashboard (which powers the "Dev Panel" network inspector) was formatting and escaping JSON for display using regex string replacement. Crucially, its regex (`/("(?:\\.|[^"\\])*")\s*:/g` and `/:\s*("(?:\\.|[^"\\])*")/g`) only targeted string values that were dictionary keys or dictionary values. It entirely skipped strings within arrays (e.g. `["<script>alert(1)</script>"]`), causing `JSON.stringify` to pass raw, unescaped output into `.innerHTML` when injecting the dev entry payloads, resulting in a critical Stored XSS vulnerability in the admin area.
**Learning:** You cannot reliably rely on regex replacement to simultaneously parse and selectively HTML-escape specific segments of a serialized JSON string. Any string component that fails to match the regular expression will fall through without being escaped, leading to an XSS vector.
**Prevention:** When dealing with functions that modify HTML structure (like syntax highlighters), the foundational input should ALWAYS be completely HTML-escaped *first*, ensuring no malicious characters can fall through. The syntax highlighting regular expressions must then be adapted to operate on the pre-escaped HTML text (e.g., using `&quot;` instead of `"`) rather than performing the escaping inline.

## 2024-05-31 - [Stored XSS via innerHTML and inline string concatenation in inline event handlers]
**Vulnerability:** Inline JS event handlers (e.g. `onclick`) in `handler/dashboard.go` were directly interpolating dynamically generated variables (like `host`, `service`, `t.id`) via string concatenation (e.g. `onclick="setServiceStatus(' + escHtml(host) + ')"`). Using `escHtml` and `encodeURIComponent` does not prevent escaping the string context within the `onclick` attribute itself if attackers supply payloads like `')-alert(1)-('`, enabling Cross-Site Scripting (XSS).
**Learning:** Browsers unescape HTML attributes before evaluating their content. Placing variables directly into JavaScript execution contexts within an HTML attribute via string concatenation bypasses regular string escaping/encoding boundaries.
**Prevention:** Always delegate dynamic data meant for JS execution to `data-*` attributes (e.g., `data-host="safeValue"`) and retrieve it using `this.getAttribute('data-host')` inside the inline handler, keeping the JavaScript context entirely separate from the templated variables.

## 2025-02-21 - [Missing secure and samesite attributes on client-side cookies]
**Vulnerability:** The client-side JavaScript in `handler/dashboard.go` was setting the `_logged_out` cookie without the `secure` and `samesite=strict` attributes (e.g. `document.cookie = '_logged_out=1;path=/';`). This exposed the cookie to transmission over unencrypted connections and increased the risk of CSRF attacks.
**Learning:** Even though the server-side might correctly set secure flags on cookies, client-side scripts manipulating those same cookies or creating new ones must also explicitly include these security attributes.
**Prevention:** Always append `;secure;samesite=strict` when assigning to `document.cookie` in client-side JavaScript, unless explicitly required otherwise for functional reasons.
