# Test Environment

Back to the [Documentation Index](../README.md)

## Overview

`testenv` is a full local lab:

- MariaDB
- Icinga2
- Icinga Web 2
- Prometheus
- Grafana
- IcingaAlertForge

## Start and Reset

Start the stack:

```bash
docker-compose -f testenv/docker-compose.yml up -d --build
```

Stop it and remove persisted state:

```bash
docker-compose -f testenv/docker-compose.yml down -v
```

Use a full reset when:

- you changed provisioning files
- you want a clean restart with no old alert objects
- you want to test host auto creation from scratch

## Managed Service Cleanup

If you only want to delete managed IAF services and keep the stack running:

```bash
testenv/scripts/purge_host_services.sh
testenv/scripts/purge_host_services.sh --apply --managed
```

The script talks to the bridge admin API and is safer than guessing directly against Icinga objects.

## Routing In The Lab

Current defaults:

- Grafana uses `test-key-grafana-local` and routes to `a-dummy-dev`
- manual and script testing uses `test-key-script-dev` and routes to `b-dummy-device`
- Team A notifications go to `alpha,omega`
- Team B notifications go to `beta,ceta`

The current lab configuration lives in [`.env.test`](../../testenv/.env.test).

## Synthetic Alerts

The lab provisions a flapping rule set generated by [`testenv/scripts/generate_flapping_alert_rules.sh`](../../testenv/scripts/generate_flapping_alert_rules.sh).

Current defaults:

- 50 synthetic devices
- one state flip every 60 seconds
- a full cycle every 120 seconds
- a 2 second phase shift between devices
- mixed `critical` and `warning`

Generated rules live in [`testenv/grafana/provisioning/alerting/flapping-device-rules.yml`](../../testenv/grafana/provisioning/alerting/flapping-device-rules.yml).

Grafana contact points live in [`testenv/grafana/provisioning/alerting/contact-points.yml`](../../testenv/grafana/provisioning/alerting/contact-points.yml).

## Icinga Notification Rules In The Lab

The Icinga image in `testenv` ships dedicated config files for users and notification rules:

- [`testenv/icinga2/conf.d/users.conf`](../../testenv/icinga2/conf.d/users.conf)
- [`testenv/icinga2/conf.d/notifications.conf`](../../testenv/icinga2/conf.d/notifications.conf)

These rules read `host.vars.notification.*`, so the test environment behaves the same way as the bridge documentation describes.

## Capturing Beauty Panel Screenshots

The lab already contains enough live data to produce screenshots of the current panel.

Recommended workflow:

```bash
docker-compose -f testenv/docker-compose.yml up -d --build
```

Then open:

- `http://localhost:9080/status/beauty`
- `http://localhost:9080/status/beauty?admin=1`

Wait until:

- the synthetic Grafana rules begin to fire and resolve
- `/admin/services` shows managed services
- the recent alerts and recent errors blocks are populated

Then capture the browser view you want.
